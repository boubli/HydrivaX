Security Guidelines

By default, HydrivaX has a drastically reduced attack surface due to its stripped-down nature. This document outlines how to maintain that security posture.

Update Process & Patch Policy

HydrivaX inherits its core user-space packages from Debian 12 (Bookworm). We guarantee 1:1 security patch parity with Debian upstream.

  • System Updates: Running hx-update --full will fetch the latest security patches. We recommend running this weekly or deploying the auto-update module.
  • Kernel Updates: When running as a Proxmox LXC container, HydrivaX uses the host's kernel. Ensure your Proxmox VE host is kept up to date. For VM and Bare Metal users, kernel updates are handled via hx-update.

Module Verification

Every module deployed via hx-deploy is checksum-verified before installation. The module definitions are hosted in the official HydrivaX repository. We do not support third-party, unsigned module repositories natively to prevent supply-chain attacks.

User Permissions

By default, HydrivaX drops you into a root shell. This is intentional for initial setup and because Proxmox LXC containers often operate entirely under unprivileged container uids.

Recommendations:

  • Deploy the sudo module: hx-deploy sudo to create a non-root user for daily administration.
  • Disable Root SSH: If you install the openssh-server module, edit /etc/ssh/sshd_config to set PermitRootLogin no after creating your user.

Firewall Recommendations

HydrivaX does not ship with an active firewall by default to prevent blocking Proxmox web UI access or cluster communication.

Hardening Steps:

  • Deploy UFW: hx-deploy ufw
  • Set default deny: ufw default deny incoming
  • Allow SSH (if installed): ufw allow ssh
  • Enable the firewall: ufw enable

Container Security Practices

If you are deploying HydrivaX inside Proxmox LXC, we strongly recommend deploying as an Unprivileged Container. This maps the container's root user (uid 0) to a non-root user (uid 100000) on the Proxmox host, providing exceptional isolation.

Only use Privileged Containers if you require direct hardware passthrough (e.g., passing a physical GPU or ZFS dataset into the container).
