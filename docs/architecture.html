System Architecture

HydrivaX is designed with a radically minimal core, where every piece of functionality is an isolated module.

1. Core System Structure

The HydrivaX core rootfs contains only systemd, bash, basic coreutils, and our custom hx-* toolchain. Everything else is removed to achieve a sub-100MB compressed template.

graph TD A[Proxmox VE / Hypervisor] --> B(HydrivaX Core OS) B --> C{hx-deploy} B --> D[Systemd Init] C --> E[Module: Networking] C --> F[Module: Docker] C --> G[Module: Database]

2. Module Deployment System

HydrivaX relies on hx-deploy rather than standard apt-get commands for user interaction. When you deploy a module, hx-deploy fetches the definition from the HydrivaX catalog, resolves dependencies dynamically, configures the service, and enables it automatically.

3. Service Lifecycle

Services deployed via modules are managed strictly by systemd. However, HydrivaX wraps standard systemd calls in its own tools to ensure permissions and dependencies don't drift. For instance, running hx-deploy remove nginx will automatically stop the service, disable it, flush the configuration, and prune orphaned dependencies.

sequenceDiagram User->>hx-deploy: hx-deploy docker hx-deploy->>Module Catalog: Fetch Docker definitions Module Catalog-->>hx-deploy: Dependencies: runc, containerd hx-deploy->>APT: Install packages silently hx-deploy->>Systemd: Write daemon.json, enable & start docker.service Systemd-->>User: Success (Module active)

4. Update Process

Because HydrivaX is heavily stripped, standard Linux kernel updates inside the LXC container are bypassed. To update, simply run hx-update. This will update the module catalog and pull the latest user-space security patches from the Debian upstream without modifying your custom configurations.
